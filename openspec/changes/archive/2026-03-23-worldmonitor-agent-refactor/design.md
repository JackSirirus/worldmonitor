# Design: WorldMonitor Agent 重构

## Context

### 背景与现状

WorldMonitor 是一个实时全球新闻监控平台，当前架构：
- **前端**: React + TypeScript + Vite，45+ 组件，Deck.gl 地图
- **后端**: Vercel Serverless Functions + Railway
- **数据**: RSS 聚合 + Redis 缓存 (Upstash)
- **AI**: Groq / OpenRouter 摘要生成

### 问题与约束

1. **Vercel 限制**: Serverless 函数执行时间 10-30 秒，无法完成 RSS 收集、AI 摘要等耗时任务
2. **状态分散**: 各组件独立管理状态，缺乏统一管理
3. **实时性弱**: 依赖轮询，无 WebSocket 推送
4. **任务不可控**: 无法追踪任务进度、查看日志、失败重试

### 目标用户

- 地缘政治观察者
- 科技创业者
- 投资者

## Goals / Non-Goals

### Goals

1. 构建完整的 Agent 框架，支持自动化新闻收集、翻译、报告生成
2. 实现 PostgreSQL 数据持久化，支持 2 个月新闻 retention
3. 引入 WebSocket 实时通信，任务状态、日志、进度实时推送
4. 实现 Zustand 状态管理，统一前端状态
5. Docker 部署，支持长时间运行的 Agent 任务
6. 完整的任务调度系统（定时 + 手动），支持重试机制

### Non-Goals

1. **用户系统**: 暂不接入用户登录/认证
2. **API 限流**: 暂不实现接口限流
3. **多租户**: 单实例部署，暂不支持多用户隔离
4. **移动端 App**: 暂不考虑原生移动端
5. **实时协作**: 暂不支持多用户实时协作编辑

## Decisions

### D1: 部署架构 - Docker 自托管

**决定**: 弃用 Vercel + Railway，迁移到 Docker 自托管

**理由**:
- Vercel Serverless 无法支持长时间运行的 Agent 任务
- Docker 可以自由控制进程生命周期
- 统一开发与生产环境

**备选方案**:
- Kubernetes: 过度复杂，适合大规模部署
- Railway 原生 Docker: 可行但成本较高

---

### D2: 数据库 - PostgreSQL

**决定**: 使用 PostgreSQL 存储新闻、报告、任务等数据

**理由**:
- 支持复杂查询（新闻聚类、分类统计）
- 支持 JSON 类型（灵活存储元数据）
- 成熟的生态与可靠性

**备选方案**:
- SQLite: 轻量但不支持复杂查询
- MongoDB: JSON 友好但聚合不如 PostgreSQL
- 继续使用 Redis: 不适合持久化大量数据

---

### D3: 任务队列 - Bull + Redis

**决定**: 使用 Bull 任务队列（基于 Redis）管理 Agent 任务

**理由**:
- Bull 是 Node.js 最成熟的任务队列方案
- 支持延迟任务、重试、优先级
- 内置进度追踪

**备选方案**:
- AWS SQS: 需要云服务，复杂度高
- RabbitMQ: 更强大但配置复杂
- 自建队列: 开发工作量大

---

### D4: 实时通信 - WebSocket 统一架构

**决定**: 构建统一的 WebSocket 消息架构，复用于 Agent、日志、数据推送

**理由**:
- Agent 执行需要实时状态反馈
- 日志需要流式输出
- 新闻更新需要即时推送
- 统一协议减少开发成本

**消息类型设计**:
```typescript
type WSMessageType = 'agent' | 'task' | 'data' | 'system';
// agent: 任务执行状态
// task: 进度更新
// data: 数据推送
// system: 心跳、系统通知
```

**备选方案**:
- SSE (Server-Sent Events): 单向，需要轮询
- 长轮询: 效率低
- 第三方 Pusher: 成本 + 依赖

---

### D5: 状态管理 - Zustand

**决定**: 引入 Zustand 作为前端状态管理方案

**理由**:
- 极简 API，无需 Provider 包裹
- 支持选择性订阅，减少不必要渲染
- 可绕过 React 直接修改状态（适合 WebWorker）
- 轻量级，适合当前项目规模

**备选方案**:
- Redux Toolkit: 功能强大但冗余
- Jotai: 原子化，适合复杂状态
- React Context: 功能有限

---

### D6: 日志系统 - Pino

**决定**: 使用 Pino 作为结构化日志库

**理由**:
- 高性能（比 Winston 快 10 倍）
- JSON 格式，便于日志收集
- 支持多输出（文件、stdout、云服务）

**备选方案**:
- Winston: 功能全但重
- Bunyan: 已停止维护
- console.log: 无法满足生产需求

---

### D7: 错误处理 - 指数退避重试

**决定**: 实现指数退避重试机制，最大 3 次

**重试配置**:
| 任务类型 | 重试次数 | 基础延迟 |
|---------|---------|---------|
| RSS 收集 | 3 | 2s |
| AI 摘要 | 2 | 5s |
| 报告生成 | 2 | 10s |
| 外部 API | 3 | 2s |

**延迟公式**: `min(baseDelay * 2^attempt, maxDelay)`

**备选方案**:
- 固定重试: 不够智能
- 线性退避: 效果不如指数

---

### D8: 数据保留策略

**决定**:
- 新闻: 2 个月
- 报告: 1 个月
- 任务日志: 30 天
- 应用日志: 7 天
- 错误日志: 90 天

**实现**: 定时任务每日检查并清理过期数据

**备选方案**:
- 永久保留: 存储成本高
- 压缩归档: 增加复杂度

---

### D9: 备份策略 - 每日自动备份

**决定**: 每日凌晨 3 点自动备份，上传到云存储，保留 7 天

**备份流程**:
1. pg_dump 导出完整数据库
2. gzip 压缩
3. 上传到云存储 (AWS S3 / 阿里云 OSS)
4. 清理 7 天前的备份

**备选方案**:
- 手动备份: 不够可靠
- 主从复制: 成本高

---

### D10: 安全架构 - Nginx 反向代理

**决定**: Nginx 作为反向代理，Docker 内网隔离

**架构**:
```
用户 → Nginx (公网) → Docker 内网 → 后端服务
```

- Nginx: HTTPS 终止、静态资源服务、WebSocket 升级
- 后端: 仅监听容器内网，不暴露端口
- API: 无需认证，依赖网络隔离

**备选方案**:
- API Gateway: 过度设计
- Cloudflare: 额外成本

---

## Risks / Trade-offs

### R1: 数据迁移风险

**风险**: 从 Redis 迁移到 PostgreSQL 可能丢失历史数据

**缓解**:
- 先同步新数据，旧数据按需迁移
- 保留 Redis 作为缓存层
- 迁移期间并行运行双系统

---

### R2: WebSocket 连接数限制

**风险**: 单实例 WebSocket 连接数有上限

**缓解**:
- 使用 Redis Adapter 支持多实例
- 前端实现断线重连
- 心跳保活检测连接状态

---

### R3: 任务队列可靠性

**风险**: Redis 故障导致任务丢失

**缓解**:
- Bull 支持任务持久化
- 监控 Redis 健康状态
- 任务结果定期清理

---

### R4: 开发体验 vs 生产环境

**风险**: Docker 部署后开发调试不便

**缓解**:
- 开发模式保持 npm run dev
- Docker 仅用于生产部署
- 使用 docker-compose 开发环境

---

### R5: Agent 执行超时

**风险**: Agent 任务可能运行超过预期时间

**缓解**:
- 任务设置最大执行时间
- 进度定期保存到数据库
- 超时自动标记失败并重试

---

## Migration Plan

### 阶段 1: 基础设施 (1 周)

1. 准备云服务器
2. 安装 Docker + Docker Compose
3. 部署 PostgreSQL + Redis
4. 配置 Nginx 反向代理

### 阶段 2: 核心数据 (1-2 周)

1. 设计并创建数据库表结构
2. 实现新闻存储 API
3. 数据迁移脚本开发
4. 清理任务开发

### 阶段 3: Agent 框架 (2 周)

1. 实现 Agent 核心框架
2. 集成 Bull 任务队列
3. 开发 WebSocket 服务
4. 实现重试机制

### 阶段 4: 前端改造 (1-2 周)

1. 引入 Zustand
2. 开发 WebSocket 客户端
3. 重构 UI 布局
4. Agent 控制面板开发

### 阶段 5: 完善 (1 周)

1. 日志系统接入
2. 备份脚本部署
3. 监控告警配置
4. 测试与调优

### 回滚策略

- 数据库: 使用 pg_dump 备份恢复
- 前端: Vercel 部署保留，可快速回切
- Docker: 保留旧镜像标签，可快速回滚

---

## Open Questions

1. **Q1**: 是否需要支持任务取消功能？
   - 当前设计支持手动触发，但不支持取消
   - 建议: 后续迭代添加

2. **Q2**: 是否需要支持任务优先级？
   - 当前所有任务优先级相同
   - 建议: 后续迭代添加

3. **Q3**: 日志是否需要接入第三方日志服务？
   - 当前存储在本地文件
   - 建议: 评估日志量后决定

4. **Q4**: 是否需要 HTTPS 证书自动续期？
   - 使用 Let's Encrypt
   - 建议: 配置 certbot 自动续期
