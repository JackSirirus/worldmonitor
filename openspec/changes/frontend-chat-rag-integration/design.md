## Context

前端 ChatWindow 调用 `POST /api/ai/chat` 与 AI 对话。当前实现仅将用户消息透传到 AI API，AI 无法回答"最新的AI新闻有哪些"这类需要实时数据的问题。

**现状**：
- 用户问题 → 直接转发 → AI API → 回答（无上下文）

**目标**：
- 用户问题 → 检索新闻 → 注入上下文 → AI API → 上下文感知的回答

## Goals / Non-Goals

**Goals:**
- 让 AI 对话能够基于本地新闻数据回答用户问题
- 用户询问最新新闻、热点分析时能返回准确结果
- 复用现有 `getNews` 函数的搜索能力
- 保持前端 API 格式兼容

**Non-Goals:**
- 不实现复杂的向量相似度检索（使用关键词搜索即可）
- 不修改前端 ChatWindow 组件
- 不实现对话历史记忆（单轮问答）
- 不实现流式响应

## Decisions

### Decision 1: 在 API 层实现 RAG，而非独立服务

**选择**：在 `server/routes/ai-chat.ts` 中直接实现新闻检索和上下文注入

**理由**：
- 改动范围小，不需要新增服务
- 复用现有的 `getNews` 函数
- 简单够用，符合当前需求

**替代方案**：
- 独立 RAG 服务：过度设计，增加运维复杂度
- MCP 工具包装：MCP 是给 AI 客户端用的，不适合前端 HTTP 调用

### Decision 2: 检索策略 - 关键词 + 时间权重

**选择**：使用用户消息作为搜索词，检索过去 7 天内的新闻

**理由**：
- 实现简单，直接复用 `getNews({ search: userMessage, fromDate })`
- 7 天窗口覆盖大多数新闻查询场景
- 避免检索过于久远的新闻

**替代方案**：
- 意图识别 + 专业查询：需要额外 ML 模型，过度复杂
- 向量检索：需要 embedding 模型，当前不需要

### Decision 3: 上下文格式 - 标题列表

**选择**：注入新闻标题列表，而非完整内容

**理由**：
- 新闻标题已包含关键信息
- 控制 token 消耗（AI API 按 token 计费）
- 减少响应延迟

**注入格式**：
```
基于以下最新新闻回答用户问题：
1. [新闻标题1] - 来源: xxx
2. [新闻标题2] - 来源: xxx
...
```

### Decision 4: 检索数量 - 最多 10 条

**选择**：限制注入最多 10 条新闻

**理由**：
- 10 条标题约 500-800 tokens，合理范围
- 足够覆盖主要新闻又不至于信息过载
- 保持响应速度

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| AI 回答包含不存在的新闻细节 | 提示 AI 仅基于提供标题回答，不要臆测 |
| 检索不到相关新闻时 AI 仍尝试回答 | 当无新闻结果时，明确告知用户"未找到相关新闻" |
| token 消耗增加导致成本上升 | 限制 10 条标题，控制 max_tokens |
| Groq API 403 错误导致无响应 | MiniMax 作为 fallback 已配置 |
